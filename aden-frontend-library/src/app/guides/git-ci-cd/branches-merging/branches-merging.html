Imagine you had a copy of your entire project where you could break absolutely everything, while the original remains safe and untouched in the cabinet. That's exactly what branches are.

In the past, I used to build new features directly into the main code. When an urgent bugfix came along in the middle, I had a problem: my code was a mix of unfinished construction site and stable parts. Today, the main-branch is sacred to me.

In this guide, I show you how I work with branches to tinker on multiple ideas simultaneously without stress and eventually merge them cleanly back into the main project.

3. Creating & Switching Branches

In the past, I had to create a branch first and then switch to it separately. Today I almost only use this one "short-command". It creates the new reality and beams me there immediately.

4. Worlds Converge: The Merge

When my feature is finished, the big moment comes: I bring the changes back home. Important to understand: A merge always happens into the branch you're currently on.

5. Merge is Not the Same as Merge

Depending on the situation, Git decides for itself how a merge is performed. The result looks different, but in both cases it's correct.

Fast-Forward

Fast-Forward Merge

The target branch hasn't changed since the branch-off. Git simply moves the pointer forward ‚Äì no new commit is created.
